在浏览器上输入URL发生什么事

置顶 | 发表于 2019-01-23 | 更新于: 2019-01-23 | 分类于 Linux网络
字数统计: 5k | 阅读时长 ≈ 17

一、为主机申请IP:DHCP态主机配置协议

  1. 假设主机最开始并没有连接网络，所以主机需要获取IP地址以及其他信息，那么就需要先使用DHCP来获取。

  2. 主机生成一个DHCP请求报文，并将这个报文放入具有目的的端口号67和源端口号68的UDP报文中。

  3. 该报文文段被放入在一个具有广播IP目的地址（255.255.255.255）和源IP地址（0.0.0.0）的IP数据报中

  4. 包含DHCP请求报文的IP据报则被放置在MAC帧中。这个以太网帧具有目的地址MAC地址FF.FF.FF.FF.FF.FF使该帧将广播到交换机。

    | 协议层 | 数据 |
    | —— | ———— |
    | 应用层 | DHCP请求报文 |

    | 传输层 | UDP
    源端口：68 （ DHCP客户端的固定端口）

    目的端口：67 （DHCP服务器的固定端口） |
    | 网络层 | 源IP地址:0.0.0.0（网络号代表这个局域网）
    目的IP地址：255.255.255.255（广播IP） |
    | 数据链路层 | 源MAC地址：主机MAC地址
    目的MAC地址：FF.FF.FF.FF.FF.FF (广播帧) |

  1. 以太网帧被发送到交换机 （他是怎么找到交换机的呢？你的网线连着呢呀）、交换机修改转发表（switch table）记录主机的MAC （自学习：每接收到一个帧都记录该帧的MAC和到达的接口），然后广播道所有与交换机相连的设备。

    | 地址 | 接口 | 时间 |
    | ———– | —- | —- |
    | 主机MAC地址 | 1 | … |

  • 以上过程就是将DHCP请求报文一层一层做封装（添加了UDP报头，IP报头字段，数据帧报头字段） 发送个交换机，
  • 交换机在其转换表中记录了一条关于主机发送的信息，然后再广播到所有与它连接的设备上去。

  1. 连接在交换机的DHCP服务器收到广播帧之后，不断向上分解得到IP数据报，UDP数据段，DHCP请求报文 ，之后DHCP服务器生成DHCP,ACK报文。DHCP服务器能够以CIDR块的形式（服务器IP子 按位与 服务器子网掩码获取子网地址的范围，并取地址分配给主机IP地址）

    该报文包含以下信息：

    • IP地址
    • DNS服务器得IP地址
    • 默认网关路由器的IP地址（第一跳的IP地址）
    • 子网块==子网掩码

    | DHCP ACK报文 | |
    | —————————— | ————————- |
    | IP地址 | 主机IP |
    | DNS服务器IP地址 | 后续进行DNS域名解析时使用 |
    | 默认网关路由器（第一跳）IP地址 | 主机的第一跳IP地址 |
    | 子网块（子网掩码） | |

  1. 将ACK报文向下进行依次封装成UDP报文段，IP报文，MAC帧,这个以太网帧中目的地址为主机MAC地址，源地址为路由器中的DHCP服务器地址。

    | 协议层 | 数据 |
    | ———- | ———————————————————— |
    | 应用层 | DHCP ACK报文（响应） |
    | 传输层 | UDP
    源端口： 67 (DHCP服务器的固定端口）
    目的端口：68 （（ DHCP客户端的固定端口） |
    | 网络层 | 源IP地址:0.0.0.0（网络号代表这个局域网）
    目的IP地址：255.255.255.255（广播IP） |
    | 数据链路层 | 源MAC地址：主机MAC地址
    目的MAC地址：FF.FF.FF.FF.FF.FF (广播帧) |

  • 得到的MAC帧进行解包，分用处理，然后应用层DHCP服务器给出响应ACK报文再次向下封装发送回个给交换机上，其中包含的内容如上。
  1. 包含DHCP ACK报文的MAC帧由路由器发送给交换机。交换机是自学习的，并且先前从主机收到的包含DHCP请求的MAC帧，所以交换机知交换机表中查询到，该向那个接口发送该MAC帧。
  1. 主机接收到包含DHCP的MAC帧，从连路层层层向上解包分用，IP报文，UDP报文段，从UDP报文段中抽取DHCP ACK报文。此时主机的DHCP客户端记录其中的IP地址和DNS服务器的IP地址，在主机的IP转发表(路由表)中安装默认网关地址。此时，主机已经初始化所有网络组件，并且准备开始处理web页面的获取。

二、DNS域名系统&查找默认网关路由器MAC地址：ARP地址转换协议

  1. 主机通过浏览器生成一个TCP套接字，套接字用于向目的域名发送一个HTTP请求。为了生成，需要知道网站的域名对应的IP地址。使用DNS服务提供这种域名到IP的互相转换。

  2. 主机生成了一个DNS查询报文，将字符串（域名）放入DNS报文的问题段中。这个DNS报文则放置在一个具有53号（DNS服务）目的端口的UDP报文中。该UDP报文段则放入目的IP地址（之前DHCP服务器ACK报文中返回的DNS服务器地址）和源IP地址（主机地址）的IP数据报当中。该IP数据报又被放置在MAC帧中，查看主机路由表，DNS服务器所在网络并不在路由表中记载，目前需要先跳到默认网关路由器上去，但是只知道默认网关路由器的IP地址，不知道默认网关路由器的MAC地址，所以需要ARP地址转换协议提供默认网关路由器的IP到MAC地址的地址转换服务。

  • 此时先要对URL网址进行DNS域名解析得到网站的IP地址，但是要给DNS前需要发送到默认网关路由器地址出去（第一跳），发送MAC帧目前并不知道默认网关的MAC地址，只知道默认网关的IP地址（之前DHCP服务器ACK报文中返回的默认网关路由器地址）

  • 所以需要先进行DNS服务器MAC地址获取的工作，要是使用ARP地址转换协议。

    | 协议层 | 数据 |
    | ———- | ———————————————————— |
    | 应用层 | DHCP ACK报文（响应） |
    | 传输层 | TCP
    源端口：
    目的端口： |
    | 网络层 | 源IP地址:主机IP地址 目的IP地址：DNS服务IP地址 |
    | 数据链路层 | 源MAC地址：主机MAC地址
    目的MAC地址：FF.FF.FF.FF.FF.FF (广播帧)
    帧类型：0806（ARP请求）
    硬件类型：1（以太网）
    协议类型：0X0800(要转换的类型为IP类型)
    硬件地址长度：6字节（对于MAC地址而言）
    协议地址长度：4字节（对于IP地址而言）
    OP:1(为IP请求) （2为应答）
    发送端以太网地址：主机MAC地址
    发送端IP地址：主机IP地址
    目的以太网地址：FF.FF.FF.FF.FF.FF (广播帧)
    目的IP地址：默认网关地址 |

    注：ARP只为同一子网上的主机和路由器接口解析IP地址。DNS为因特网中的任何的主机解析为IP地址。

  1. 主机生一个具有目的IP地址（默认网关）的ARP查询报文，将这个ARP查询报文放置在一个广播目的地址（FF.FF.FF.FF.FF.FF）的MAC帧中，并向交换机发送这个MAC帧，交换机将这个MAC帧广播给所有与它连接的设备上去（包括网关路由）。

    • 在发送ARP请求的时候是广播（广播请求），在发送ARP应答的时候是并不是群发，而是根据主机的IP,MAC地址单播的（单播回应）

  2. 默认网关路由器接收到这个MAC帧，提取得到ARP报文，填写源MAC地址（默认网关路由器的MAC地址），然后根据主机IP（目的IP地址），主机MAC地址（目的MAC地址）进行单播发送ARP应答。

  3. 主机接收到后对ARP进行缓存（ARP表），MAC和IP的映射。该缓存有生命周期一般为20分钟。20分钟后操作系统重新对ARP缓存表操作。此时主机就知道了默认网关路由器的MAC地址。

​

IP地址 MAC地址 TTL(TIME TO LIVE)
默认网关路由器IP地址 默认网关路由器MAC(ARP回应中的源MAC地址字段) 20

万事具备了。现在主机能够使包含DNS查询报文的MAC帧寻址到默认网关路由器的MAC地址（第一跳）了。（继续进行DNS解析）

三、查找目的域名的IP:DNS域名系统

  1. 默认网关路由器接收到该帧并抽取DNS查询的IP数据报，并根据转发表（路由器表）决定该 IP 数据报应该转发的路由器，直至到达服务器所在局域网路由器
  2. 因为路由器具有内部网关协议（RIP路由信息协议、OSPF开放最短路径优先协议）和外部网关协议（BGP边界网关协议/以及因特网的域间协议BGP所填写）这两种路由选择协议，因此路由表中已经配置了网关路由器到达 DNS 服务器的路由表项。
  3. 到达 DNS 服务器之后，DNS 服务器抽取出 DNS 查询报文，并在 DNS 数据库中查找待解析的域名。
  4. 找到 包含这个主机名的IP地址映射的DNS回答报文之后，发送 DNS 回答报文，将该回答报文放入 UDP 报文段中，然后放入 IP 数据报中，通过路由器反向转发回网关路由器，并经过以太网交换机到达主机。
  5. 主机从DNS报文中取得网占的IP地址。最终，在上面进行大量工作后，主机准备和该网站服务器建立连接。

四、Web客户-服务器交互：TCP和HTTP

  1. 既然主机已经有了网站服务器的IP地址，则他现在可以生成TCP套接字，这个套接字用于向发送HTTP GET报文（长连接）。但是TCP是需要连接的，要跟网站服务器之间进行3次握手来建立连接。此时主机生成一个具有目的端口80的TCP SYN报文该TCP报文段 放在具有目的IP地址为域名解析后的网站服务器IP地址，数据报则放置在MAC地址为默认网关路由器的帧中，并向交换机发送该MAC帧。
  1. （第一次握手）主机向目的服务器发送的连接请求报文，SYN = 1,ACK = 0,.先择一个初始序号为X。主机的状态机变为SYN_SENT。（确认客户端的发送能力，
  1. （第二次握手）目的服务器收到连接请求报文段后，如果同意建立连接，则向主机发送一个连接请求报文段，SYN = 1 ACK = 1,确认序号为X+1.同时也选择一个初始序号Y。服务器的状态机由LISTEN变为SYN_RCVD（确认服务器的接收能力，确认服务器的发送能力）

  1. （第三次握手）主机收到目的服务器的连接确认报文后，好要向目的服务器发出确认，确认序号为Y+1,序号为X+1。（确认客户端的接受能力）

    • 三次握手是TCP可靠性的重要表现，《计算机网络》第四版中说三次握手是为了防止已失效的连接请求报文段突然又传送到了服务端，因而产生错误。
    • 客户端发送的连接请求如果在网络中滞留，那么就会隔很长一段时间才能收到服务器端发回的连接确认。客户端等待一个超时重传时间之后，就会重新请求连接。但是这个滞留的连接请求最后还是会到达服务器，如果不进行三次握手，那么服务器就会打开两个连接。如果有第三次握手，客户端会忽略服务器之后发送的对滞留连接请求的连接确认，不进行第三次握手，因此就不会再次打开连接
  • 但是假设不进行三次握手，只要服务器发出请求客户端回应就连接成功的化，但是一旦服务器的确认报文在发送过程中丢失了，此时服务器端认为连接已经建立，就会维护这个连接的，维护这个连接是需要花费成本的，但是服务器维护的连接并不是双方认可的连接。
  • 如果服务器的响应报文一直丢失，客户端一直认为连接失败，一直发送连接请求。此时服务器就有了大量的闲时连接，导致服务器资源不够，服务器谈话（给攻击服务器的人提供了一种返方法）处于安全服务器考虑，将最后一个连接请求放到客户端去，即使最后一次报文丢失，也是子客户段方。

  1. 连接建立完成后，服务器会浏览器响应一个301永久重定向，这样浏览器访问“http://www./XXXX.com”而非“ http://XX.com"。这个原因跟搜索引擎排名有关。如果一个页面有两个地址，就像http://www.igoro.com/ 和http://igoro.com/，搜索引擎会认为它们是两个网站，结果造成每一个的搜索链接都减少从而降低排名。而搜索引擎知道301永久重定向是 什么意思，

    • 这样就会把访问带www的和不带www的地址归到同一个网站排名下。
    • 还有就是用不同地址会造成缓存友好性变差。当一个页面有好几个名字时。他就会在缓存里出现好几次。

  2. 浏览器知道了永久重定向地址后，他就发送确认数据报。（确认应答机制）

  3. 服务器确认收到后，客户此时终于可以发送HTTP GET报文了，将请求页面的内容放入请求报头当中，并将报文端发送到TCP套接字中层层封装，数据帧的形式发送个服务器，服务器拿到数据帧，向上一步一步解包分用。

  4. 服务器生成一个Http响应报文，也层层封装，发送个客户端。这样你来我往的（确认应答机制），客户端在收到的HTTP响应报文后，抽取Web页面内容后，进行渲染，显示Web页面

五、页面渲染<前端待学习>

现代浏览器渲染页面的过程是这样的：解析 html 以构建 DOM 树 –> 构建渲染树 –> 布局渲染树 –> 绘制渲染树。

渲染树是在 DOM 树中加入 css 或 html 中的 style 样式而形成。渲染树只包含需要显示在页面中的 DOM 元素，像 <head> 元素或 display 属性值为 none 的元素都不在渲染树中。在浏览器还没接收到完整的 html 文件时，它就开始渲染页面了。

　　在遇到外部链入的脚本标签、样式标签、图片时，会再次发送 http 请求重复上述的步骤。在收到 css 文件后，会对已经渲染的页面重新渲染，加入它们应有的样式。图片文件加载完，立刻显示在相应位置。在这一过程中可能会触发页面的重绘或重排。

六、客户端断开连接（4次挥手）

​ 由于TCP连接是全双工的，因此连接双方都必须单独进行关闭。

  1. （第一次挥手）客户端发起一个FIN，用于关闭客端到服务器的数据传输，初始序号X

    • 此时客户端状态机从ESTABLSHED变为FIN_WAIT_1状态。close()方法

  2. （第二次挥手）：服务器收到这个FIN，它发会一个ACK，确认序号为X+1.初始序号Y

    • 此时服务器状态机从ESTABLSHED变为COLSE_WAIT 。read（）返回0
    • 客户端收到服务器对FIN报文的确认，则状态机从FIN_WAIT_1变为FIN_WAIT_2开始等待服务器的FIN报文。确认序号Y+1.确认序号为X+1

  3. （第三次挥手）：服务器关闭客户端连接，发送一个FIN给客户端。序号Y+2,确认序号X+1

    • 服务器的状态机从COLSE_WAIT变为LAST_ACK等待最后一个ACK的到来（服务器确认受到）。

  4. （四次挥手）：客户端接收到服务器的FIN报文后,序号为X+2 确认序号为Y+2。客户端等待2MSL

    • 此时客户端从状态机从FIN_WAIT_2变为TIME_WAIT。

    • 客户端要等待⼀个2MSL=120s(Max Segment Life,） 报⽂最⼤⽣存时间)的时间,才会进⼊CLOSED状态 。

    • 客户端是在等待，保证迟到的报文尽快收到。在此等待状态下，服务器与客户端之间的连接并没有完全断开，无法重新建立新的连接。

    • 解决允许创建端口号相同的但是IP地址不同的多个socket连接

    • 为什么建立连接时是3次握手，而建立连接时却是4次挥手

      • 3次握手时，服务的对于客户端的第一次请求中（第二次握手时）服务器将连接请求SYN和响应同时进行发送（捎带应答）
      • 而关闭连接时，当收到客户端的FIN报文时，仅仅表示客户端不再发送数据，还可以接收数据。服务器也未必已经将要发的数据全部发送完了，所以此时一次性将服务器的ACK和FIN发送就可能导致服务器要发送的书数据发送不了，所以先给客户端响应ACK后进行未发完数据发送，发送完毕后再发送FIN个客户端，表示服务器不再发送数据了。保证了TCP的可靠性。
      • 因此，服务器的ACK和FIN一般都会分开发送。

总结

  • DHCP：获取主机IP地址（局域网内广播
  • DNS&ARP：
    • 获取默认网关（第一跳）（因为DNS查询外网查询）的MAC地址（ARP）
    • 获取到默认网关地址后就可以进行封装成数据帧，利用DNS对网站的域名进行解析了。
    • 注：ARP只为同一子网上的主机和路由器接口解析IP地址。DNS为因特网中的任何的主机解析为IP地址。
    • DNS域名解析过程，获得网站的IP地址
  • Web客户-服务器交互：TCP和HTTP
    • 可以进行HTTP GET 请求了，但是TCP是面向连接的，所以需要进行3次握手
    • 建立连接完成后要进行服务器永久重定向，浏览器跟踪重定向（确认应答机制）
    • 建立连接后，就以确认应答机制你来我往的发送数据。
    • 客户端收到服务器HTTP响应报文后，进行页面渲染
  • 页面渲染<前端待学习>
    • 解析 html 以构建 DOM 树 –> 构建渲染树 –> 布局渲染树 –> 绘制渲染树。
  • 终止连接TCP4次挥手
